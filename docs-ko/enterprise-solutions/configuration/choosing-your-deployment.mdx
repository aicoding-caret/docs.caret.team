조직에 맞는 구성 방식을 선택하세요. 대부분의 팀은 빠른 도입을 위해 SaaS 구성을 먼저 선택하고, 복잡한 요구 사항이 있는 기업은 Self-Hosted 인프라를 선택합니다.

## 구성 경로

  <Card title="SaaS 공급자 구성" icon="cloud" href="/enterprise-solutions/configuration/remote-configuration/overview">
    ### 웹 콘솔 기반 빠른 설정
    
    ✅ 인프라 불필요

    ✅ 웹 기반 관리자 콘솔

    ✅ 간소화된 자격 증명 관리

    - 소규모~중간 규모 팀(5~50명)
    - 빠른 배포가 필요한 경우
    - DevOps 리소스가 محدود한 경우
    - 표준 보안 요구 사항
    - 단일 리전 배포
  </Card>
  
  <Card title="Self-Hosted 구성" icon="server" href="/enterprise-solutions/configuration/infrastructure-configuration/overview">
    ### 완전한 인프라 제어
    
    ✅ 자체 AWS/GCP/K8s  
    ✅ VPC 엔드포인트 및 프라이빗 연결  
    ✅ 멀티 계정 구성  
    ✅ 고급 컴플라이언스/감사  
    ✅ GitOps 워크플로
    
    **추천 대상:**
    - 대규모 엔터프라이즈(50명 이상)
    - 복잡한 보안 요구 사항
    - 기존 클라우드 인프라 보유
    - 멀티 리전 배포
    - 맞춤형 컴플라이언스
  </Card>
</CardGroup>

## 상세 비교

### 기능 비교

| 항목 | SaaS | Self-Hosted |
|---------|------|-------------|
| **구성 방식** | Web UI | YAML + Helm/Kubernetes |
| **인프라** | 불필요 | AWS/GCP/K8s 전체 |
| **VPC 엔드포인트** | 기본 | 완전한 프라이빗 연결 |
| **멀티 계정** | ❌ | ✅ |
| **IAM** | 표준 RBAC 역할 | 표준 RBAC 역할 |
| **컴플라이언스** | 표준 | 맞춤 프레임워크 |
| **GitOps** | ❌ | ✅ |
| **운영** | Caret 관리 | 자체 운영 |
| **업데이트** | 자동(확장) | 자동(확장) + 인프라 제어 |

### 보안 & 컴플라이언스

| 항목 | SaaS | Self-Hosted |
|------------|------|-------------|
| **네트워크 암호화** | HTTPS/TLS | HTTPS/TLS |
| **네트워크** | 공용 인터넷 | 프라이빗 VPC 엔드포인트 |
| **접근 제어** | 표준 RBAC | 표준 RBAC |
| **감사 로그** | OpenTelemetry 트레이스 | OpenTelemetry + 인프라 로그 |
| **데이터 레지던시** | Caret 관리 배포 | 고객이 배포/관리 |

### 비용 구조

| 비용 항목 | SaaS | Self-Hosted |
|---------------|------|-------------|
| **Caret 구독** | 고정 엔터프라이즈 요금 | 고정 엔터프라이즈 요금 |
| **추론 비용** | 사용량 기반 | 사용량 기반 |
| **인프라** | ✅ 불필요 | Kubernetes/네트워킹/스토리지 |
| **인력** | ✅ 불필요 | DevOps 팀 필요 |
| **총 비용** | 단순/예측 가능 | 규모에 따라 변동 |

## 마이그레이션 경로

<Note>
대부분의 조직은 빠른 배포를 위해 SaaS로 시작한 뒤, 요구 사항이 커지면 Self-Hosted로 이동합니다. 초기 위험을 낮추고 실제 사용 패턴에 맞는 인프라를 구축할 수 있습니다.
</Note>

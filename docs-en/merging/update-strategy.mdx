## 목적
- Caret upstream 변경을 캐럿 매뉴얼에 빠짐없이 반영하고, 브랜딩 차이(Caret vs Caret)를 일관되게 유지합니다.
- 네비게이션(`docs/docs.json`)과 스타일(`docs/styles.css`)을 포함해 문서 전반을 검증합니다.
- 캐럿 전용 정책/기능(모델 제한, 인증/라우팅, UI/브랜딩 차이)을 별도로 수집·주석 처리해 Caret 원문과 구분합니다.

## 준비 체크리스트
- 기준 브랜치: `caret-main`(기존) vs `caret-latest`(업스트림) diff 확보.
- 도구: `git diff caret-main..caret-latest --name-only -- docs`로 영향 파일 목록 확인.
- 브랜딩 원칙: 기능 설명은 원문을 유지하되 제품명은 “캐럿(Caret)” 병기 여부를 결정해 일관 적용.
- 컴포넌트: `<Warning>`, `<Tip>`, `<Note>`, `<Card>` 등 MDX 컴포넌트를 그대로 사용.

## 업데이트 절차
1) **변경 수집**: `git diff caret-main..caret-latest -- docs`로 변경 파일을 나열하고, 우선순위(온보딩, 주요 기능, 프로바이더, CLI)를 정합니다.
2) **내용 반영**: 각 파일을 비교해 기능 추가/제거/주의사항을 옮깁니다. 예) AGENTS.md 지원, CLI 프리뷰 범위, 모델/프로바이더 옵션.
3) **캐럿 전용 사항 주석**: Cline과 다른 정책/제한(예: 지원 모델/지역 차이, 인증 라우팅, 번들 여부)은 `<Note>`로 “캐럿 전용”을 명시합니다.
4) **브랜딩/용어**: “Caret”을 “캐럿(Caret)”으로 병기하거나, 기능 출처가 중요할 때만 Caret 표기를 유지합니다.
5) **네비게이션 확인**: 새/이동된 문서는 `docs/docs.json`에 추가하고, 섹션 순서를 유지합니다. 필요 시 `docs/styles.css` 레이아웃 영향 검토.
6) **링크/에셋 검증**: 내부 링크 경로(`/...`)와 이미지 경로(`docs/assets`)가 유효한지 확인합니다.

## 검증 및 리뷰
- 빠른 점검: 로컬에서 `npm run docs`(또는 사용 중인 문서 프리뷰 명령)로 빌드/링크 깨짐을 확인합니다.
- 텍스트 품질: 주요 섹션(설치, 시작하기, 주요 기능, 프로바이더 설정, CLI)만 우선 리뷰하고, 나머지는 후속 패스에서 정리합니다.
- 캐럿 정책 확인: 모델 지원/차단, 인증/라우팅, 번들 여부가 Caret 문구와 충돌하지 않는지 확인하고 필요 시 `<Note>`로 표시합니다.
- TODO 기록: 반영 못한 파일/링크는 워크로그에 명시하고 다음 사이클에 처리합니다.
